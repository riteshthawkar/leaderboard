# Deployment

MS-VISTA supports two deployment topologies:

- The root `Dockerfile` runs the Hugging Face single-origin deployment. Nginx serves React and proxies `/api` to a private Gunicorn process.
- `deployment/api/Dockerfile` and `frontend/Dockerfile` provide independent API and frontend containers for conventional split hosting.

In both topologies Flask remains API only. It does not render frontend routes or expose `frontend/static`.

## Production Origins

Use explicit HTTPS origins for both services. Same-site subdomains are preferred
because browser cookie policies are more reliable:

```text
Frontend: https://app.example.com
API:      https://api.example.com
```

Configure the API service:

```env
FLASK_ENV=production
DEPLOYMENT_MODE=public
SECRET_KEY=<generate-with-python-secrets-token-hex-32>
FRONTEND_BASE_URL=https://app.example.com
API_BASE_URL=https://api.example.com
OAUTH_REDIRECT_BASE_URL=https://api.example.com
CORS_ORIGINS=https://app.example.com
SESSION_COOKIE_SECURE=true
SESSION_COOKIE_SAMESITE=Lax
PRIVACY_POLICY_URL=https://app.example.com/privacy
TRUST_PROXY_HOPS=1
DISABLE_SUBMISSION_AUTH=false
```

Set `TRUST_PROXY_HOPS` to the exact number of reverse proxies between the
browser and Flask. Keep it at `0` for a directly exposed API process. An
incorrect value can either merge every user into one rate-limit bucket or trust
client-supplied forwarding headers.

If frontend and API are on unrelated sites, set
`SESSION_COOKIE_SAMESITE=None` with `SESSION_COOKIE_SECURE=true`. Some browsers
can still restrict third-party cookies, so same-site subdomains are recommended.

Do not use `CORS_ORIGINS=*`. The API rejects wildcard CORS because requests use
credentialed session cookies.

When a local `.env` file is present, startup rejects duplicate variable names
before loading any values. Keep one assignment per key so OAuth, cookie, and
origin settings cannot be silently overridden by a later line.

## Build The API

```bash
cd Combined-Leaderboard
docker build -f deployment/api/Dockerfile -t ms-vista-api .
```

Run it with persistent data:

```bash
docker run --rm -p 5050:7860 \
  -e DEPLOYMENT_MODE=local \
  -e SECRET_KEY="$(python -c 'import secrets; print(secrets.token_hex(32))')" \
  -e FRONTEND_BASE_URL=http://localhost:8080 \
  -e API_BASE_URL=http://localhost:5050 \
  -e OAUTH_REDIRECT_BASE_URL=http://localhost:5050 \
  -e CORS_ORIGINS=http://localhost:8080 \
  -e SESSION_COOKIE_SECURE=false \
  -e DISABLE_SUBMISSION_AUTH=true \
  -e GROUND_TRUTHS_SOURCE=local \
  -e GROUND_TRUTHS_DIR=/run/ground-truths \
  -v "$PWD/.data:/data" \
  -v "$PWD/Ground_truths:/run/ground-truths:ro" \
  ms-vista-api
```

The service descriptor is available at `http://localhost:5050/` and
`http://localhost:5050/api`. Readiness is available at `/api/health`.

## Build The Frontend

Build the frontend container with the browser-visible API origin:

```bash
cd Combined-Leaderboard
docker build \
  -f frontend/Dockerfile \
  --build-arg VITE_API_BASE_URL=http://localhost:5050 \
  --build-arg VITE_PRIVACY_POLICY_URL= \
  -t ms-vista-frontend \
  frontend

docker run --rm -p 8080:8080 ms-vista-frontend
```

Open `http://localhost:8080`. The SPA sends credentialed requests directly to
`http://localhost:5050/api/*`.

For a non-container build:

```bash
cd frontend
npm ci
VITE_API_BASE_URL=https://api.example.com npm run build
```

Deploy `frontend/dist/` to any static host that supports an SPA fallback to
`/index.html`. `frontend/public/_redirects` provides that fallback on compatible
platforms.

## Spatial Bundle

The checked-in spatial bundle is a demo and the API will not accept submissions
against it. Before enabling the spatial track, generate the official public
manifest and template with `evaluation/spatial_reasoning/build_server_bundle.py`. Keep the
generated administrator QA key outside source control and configure:

```env
REQUIRE_OFFICIAL_SPATIAL=true
```

The Spatial readiness check does not load private ground truth. It verifies the
official manifest, all six public condition and scoring-group counts, and the
question and template hashes before reporting the bundle healthy.

Participants upload one `spatial_reasoning_submission.zip` generated by the official
harness. The API applies a route-specific multipart limit, keeps the upload
stream in memory, and reads `submission.jsonl`, `run_manifest.json`, and
`leaderboard.json` without extracting them to disk. After validation, it stores
the exact archive and all three members in SQLite and exposes them for visible
submissions through the public evidence API. Configure the `MAX_SPATIAL_*`
limits from `.env.example` if an official release requires different bounds.

## Authentication

The frontend sends `credentials: include` and uses the CSRF token returned by
`/api/auth/me`. The API must therefore have:

- An exact `CORS_ORIGINS` entry for every frontend origin.
- `SESSION_COOKIE_SECURE=true` in HTTPS production.
- Correct `SESSION_COOKIE_SAMESITE` behavior for the chosen domains.
- `FRONTEND_BASE_URL` for password reset and post-auth redirects.
- `API_BASE_URL` for verification links.
- `MAX_AUTH_REQUEST_BYTES` for the bounded JSON auth body size; the default is 16 KB.

New passwords accept 15 to 128 characters without composition rules. Resetting
a password revokes every existing session for that account. OAuth authorization
code flows use signed state, an HttpOnly state cookie, and PKCE with S256.

OAuth callback URLs belong to the API service:

```text
https://api.example.com/api/auth/oauth/google/callback
https://api.example.com/api/auth/oauth/microsoft/callback
```

After OAuth completes, the API redirects the browser to the frontend origin.

Email/password registration and reset require Azure Communication Services
Email or SMTP. Without an email provider, authenticated production deployment is
not ready.

## Data And Scaling

For a single-instance pilot:

```env
LEADERBOARD_DATA_DIR=/data
RESULTS_DIR=/data/results
DATABASE_URL=sqlite:////data/leaderboard.db
WEB_CONCURRENCY=1
GUNICORN_THREADS=4
LIMITER_STORAGE_URI=memory://
HEALTH_CACHE_SECONDS=15
TRUST_PROXY_HOPS=0
SQLITE_BUSY_TIMEOUT_MS=5000
AUTO_BACKUP_ENABLED=true
AUTO_BACKUP_MIRROR_DIR=/offsite/ms-vista
REQUIRE_OFFSITE_BACKUP=true
AUTO_BACKUP_INTERVAL_HOURS=48
AUTO_BACKUP_RETENTION_COUNT=15
```

Mount `/data` persistently. Keep one worker while using SQLite. SQLite is an
appropriate choice for this low-concurrency deployment, but it permits only one
writer at a time and must not be placed on an ephemeral filesystem. WAL mode and
the busy timeout handle short overlaps between account and submission writes.
Public startup fails when any SQLite database is configured with
`WEB_CONCURRENCY` other than `1`.

Scheduled backups are written to `/data/backups`, validated before retention,
and exposed only through admin APIs. `AUTO_BACKUP_MIRROR_DIR` must point to a
second mounted filesystem in public mode. Each copy is written atomically,
SHA-256 compared with the primary archive, and independently checked for ZIP and
SQLite integrity. Public readiness fails when the mirror is missing, overdue, or
on the same filesystem. Public startup also refuses a missing, disabled, or
same-filesystem mirror configuration.

Run an offline recovery drill without touching the live database:

```bash
python -m backend.backup_cli verify /offsite/ms-vista/ms-vista-backup-<timestamp>.zip
python -m backend.backup_cli restore /offsite/ms-vista/ms-vista-backup-<timestamp>.zip \
  --destination /tmp/ms-vista-recovery
```

For multiple API instances, migrate to PostgreSQL and use Redis for
`LIMITER_STORAGE_URI`.

Use `/api/health/live` for process liveness and `/api/readiness` for database,
storage, auth, email, deployment, ground-truth, and backup readiness.

## Private Test Mode

For a private temporary environment only:

```env
DISABLE_SUBMISSION_AUTH=true
TEST_SUBMISSION_USER=test-deployment@ms-vista.local
ADMIN_EMAILS=
```

Anyone who can reach the API can submit under the shared identity in this mode.
Never use it for a public launch.

## Hugging Face Spaces

The root Docker image is a single-origin Docker Space on port `7860`. Nginx
serves the built frontend and proxies `/api` to Gunicorn on an internal port.
This keeps authentication cookies first party while preserving the API-only
Flask boundary.

Create two private writable Storage Buckets and mount them separately:

```bash
hf buckets create <owner>/ms-vista-data --private
hf buckets create <owner>/ms-vista-backups --private
hf spaces volumes set <owner>/ms-vista-leaderboard \
  -v hf://buckets/<owner>/ms-vista-data:/data \
  -v hf://buckets/<owner>/ms-vista-backups:/backup
```

The container refuses to start without both mounts. `/data` stores SQLite and
leaderboard state; `/backup` receives the separately verified 48-hour backup
mirror. Default Space disk is ephemeral and must not hold production data.

If private ground truth should not be committed, configure:

```env
GROUND_TRUTHS_SOURCE=hf
GROUND_TRUTHS_HF_REPO=<owner/private-ground-truth-repo>
GROUND_TRUTHS_HF_REVISION=<full-40-character-commit-sha>
HF_TOKEN=<read-token-for-the-private-repo>
```

Official answer JSONL files are ignored by Git and the Docker build context.
The production image never contains the answers. Public readiness requires an
immutable Hugging Face commit SHA and verifies exact ID parity between every
private answer key and its public question file.

The following Space secrets are required for the current configuration:

```text
SECRET_KEY
ADMIN_EMAILS
HF_TOKEN
GROUND_TRUTHS_HF_REPO
ACS_CONNECTION_STRING
ACS_SENDER_ADDRESS
MICROSOFT_CLIENT_ID
MICROSOFT_CLIENT_SECRET
MICROSOFT_TENANT_ID
```

`SPACE_HOST` is supplied by Hugging Face. The startup script derives the exact
HTTPS frontend, API, OAuth, CORS, and privacy origins from that trusted value.
For Microsoft sign in, register
`https://<space-host>/api/auth/oauth/microsoft/callback` in the Azure app.

## Production Acceptance

Before opening registration:

1. Register and verify at least one address listed in `ADMIN_EMAILS`. Public readiness remains unhealthy until this account exists.
2. Complete one Microsoft sign-in through the provider callback and confirm the resulting profile session.
3. Deliver one verification email and one password-reset email to a controlled test inbox.
4. Publish an institution-approved privacy policy and set both `PRIVACY_POLICY_URL` and frontend `VITE_PRIVACY_POLICY_URL`.
5. Mount `AUTO_BACKUP_MIRROR_DIR` on a second filesystem and complete the restore drill above.
6. Build and run both Docker images on the deployment host.
7. Run the external smoke check, omitting `--require-spatial` until that pipeline is ready:

```bash
python scripts/production_smoke.py \
  --api-url https://api.example.com \
  --frontend-url https://app.example.com
```

This checklist is intentionally outside CI/CD. It validates the real provider,
mail, storage, TLS, proxy, cookie, CORS, and frontend-header configuration.
